---

사용자가 `/makedocs`를 실행하면 현재 프로젝트를 분석하여 `docs/` 디렉토리에 문서를 생성하거나 갱신한다.

## 실행 절차

1. 프로젝트 루트에서 다음을 파악한다:
   - 서비스 목적 및 타겟 사용자 (README, 코드, 설정 파일에서 추론)
   - 기술 스택 (언어, 프레임워크, DB, 인프라)
   - 프로젝트 구조 (디렉토리 트리, 핵심 파일)
   - 개발 환경 (실행 방법, 포트, 환경변수)
   - 배포 환경 (서버, 도메인, 배포 방식)
   - CI/CD 설정 (있다면)

2. `docs/` 디렉토리가 없으면 생성한다.

3. 아래 파일들을 생성하거나 기존 내용을 갱신한다:

### docs/README.md
문서 인덱스. 다음을 포함:
- 각 문서 파일 링크 및 한 줄 설명
- 서버/인프라 핵심 정보 요약 테이블 (있는 경우)

### docs/service.md
서비스 관점의 문서. 기술이 아닌 "무엇을 왜 어떻게 제공하는가"를 다룬다. 다음을 포함:

**서비스 개요**
- 한 문장 요약 (엘리베이터 피치)
- 서비스가 해결하는 문제
- 핵심 가치 제안

**타겟 사용자**
- 주요 사용자 페르소나 (누구인가, 어떤 상황에서 쓰는가)
- 사용자의 Pain Point

**서비스 플로우**
- 사용자 여정 단계별 설명 (진입 → 핵심 액션 → 결과)
- 주요 터치포인트
- ASCII 플로우 다이어그램 (간결하게)

**핵심 기능**
- 기능명과 사용자 관점의 설명 (개발 용어 최소화)
- 각 기능이 제공하는 가치

**고객 경험 (UX)**
- 첫 인상 / 온보딩 경험
- 반복 사용 시 경험
- 이탈 방지 요소

**비즈니스 모델** (파악 가능한 경우)
- 수익화 방식
- 주요 지표 / KPI

코드베이스와 README, 설정 파일 등에서 최대한 파악하고, 파악이 어려운 항목은 합리적으로 추론하되 "(추론)" 표시를 붙인다.

### docs/architecture.md
- 기술 스택 테이블
- 서비스 흐름도 (ASCII)
- 프로젝트 디렉토리 구조
- 주요 API 엔드포인트 (있는 경우)
- DB 모델 관계 (있는 경우)

### docs/local-dev.md
- 사전 요구사항
- 최초 설치 방법
- 실행/중지 명령어
- 접속 URL 및 포트
- DB 마이그레이션 방법 (있는 경우)
- 자주 쓰는 명령어

### docs/deployment.md (배포 환경이 있는 경우)
- 서버 정보 (플랫폼, OS, IP, 도메인)
- SSL 설정
- 최초 서버 설정 절차
- 일반 배포 절차
- DB 마이그레이션 절차
- 유용한 운영 명령어

### docs/cicd.md (CI/CD가 있거나 필요한 경우)
- 자동 배포 흐름 설명
- 설정 절차 (SSH 키, Secrets 등)
- 주의사항

## 작성 원칙

- 코드블록을 적극 활용해 복붙 가능하게 작성한다
- 추측이 아닌 실제 코드에서 확인한 정보만 기재한다 (service.md 예외: 추론 허용, 단 "(추론)" 표시)
- 기존 docs/ 파일이 있으면 덮어쓰지 않고 변경된 부분만 갱신한다
- 파일 생성/갱신 후 git에 커밋할지 사용자에게 확인한다
